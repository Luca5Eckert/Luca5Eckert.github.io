  • ECST read model — Recommendation Service consumes created-book, updated-book, deleted-book, created-rating from Kafka. All query-time data is local. The constraint is a design invariant, not a performance optimization
  • Embedding pipeline — replaced 13-dim genre vectors with LangChain4j AllMiniLmL6V2 384-dim embeddings, L2-normalized before storage. ANN retrieval via HNSW index, re-ranked with 70/30 semantic/popularity blend
  • Reactive gateway — WebFlux entry point with centralized JWT validation and Redis-backed per-route rate limiting. Built from scratch rather than layering on Spring Cloud Gateway to retain direct pipeline control
  • Latency evolution — v1 used an external Python ML service (~400ms); v2 brought retrieval in-JVM with pgvector (~100ms). The gain came from eliminating the cross-process hop. v3 adds 384-dim semantic embeddings — final latency not yet published

Kairos PROTOTYPE
Knowledge graph engine — JVM-native embedding, dual-pipeline ingest, graph-guided retrieval what changes when memory has structure you didn't build?
ONNX · Neo4j · pgvector

Dual-pipeline system that builds both a semantic index and a typed knowledge graph from the same ingested content. Retrieval is modeled on HippoRAG 2 (NeurIPS 2024): vector search finds anchor nodes, Personalized PageRank propagates relevance through the graph from those seeds, RRF fuses both candidate sets. Status: prototype — architecture defined, core pipelines implemented, not production-ready.

  • Graph-guided retrieval — pgvector finds anchor nodes (vector proximity); Neo4j runs Personalized PageRank from those seeds (graph propagation); RRF fuses both candidate sets. A query about "weight updates" reaches passages about "backpropagation" through "gradient descent" even when no single passage is vector-close to the query
  • No Python sidecar — ONNX Runtime + DJL HuggingFace Tokenizers run all-MiniLM-L6-v2 directly on the JVM. The model is a file, not a service. One runtime, no cross-process overhead, no hidden coupling
  • Synonymy edges — concept nodes whose embedding vectors exceed a cosine similarity threshold are automatically linked during ingest. backprop and backpropagation are connected without manual normalization; the graph normalizes itself as it grows
  • Hexagonal architecture — pgvector, Neo4j, Gemini Flash, and ONNX each sit behind a port interface. Swapping concept extraction from Gemini Flash to a local Ollama model is an adapter change, not an application-layer change
  • Role — independent project outside WEG work; solo system design across ingestion pipelines, graph modeling, retrieval architecture, and D3.js visualization direction

OpenIT TEAM OF 3
IoT access control and payment — hardware to interface, one coherent flow how do hardware events, payment confirmation, and backend coordination become one coherent system?
WebFlux · MQTT · ESP32

Automated parking system spanning ESP32 firmware, MQTT messaging, Node-RED orchestration, reactive Spring backend, Mercado Pago payment confirmation, and a React terminal interface. Gate release is gated on confirmed payment — not on optimistic UI state.

// full system flow — entry to gate release
  • Two DDD bounded contexts — access (entry detection, gate coordination, exit logging) and payment (Mercado Pago Checkout Pro preference creation, webhook receipt, SSE delivery); each with four-layer clean architecture. Context boundaries defined before any code was written
  • Node-RED as IoT orchestrator — subscribes to MQTT device events, calls POST /access/entry on entry, and publishes gate-open commands back to ESP32 via MQTT after POST /open-gate is received from the backend. The backend does not talk to hardware directly
  • SSE for payment status — backend holds a reactive SSE connection per payment ID and pushes confirmation the moment the Mercado Pago webhook lands. No polling, no client retry loop — gate release happens on confirmed event receipt
  • Role — team project (3 people) in the training program; hands-on work across reactive backend, MQTT / Node-RED integration flow, and real-time status delivery
